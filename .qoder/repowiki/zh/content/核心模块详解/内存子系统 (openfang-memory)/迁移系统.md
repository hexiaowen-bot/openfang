## 简介
本文件原本面向 OpenFang 的迁移系统模块，系统化阐述数据库模式升级机制、数据迁移脚本管理、版本兼容性处理策略。然而，根据最新的代码变更，迁移系统已在项目中完全移除。本文档现更新为说明项目不再支持迁移功能，并提供替代方案和最佳实践。

## 详细组件分析

### 组件一：结构化存储（向后兼容与代理管理）
- 设计要点
  - 代理存储在迁移后新增列（如 session_id、identity）时，采用 ALTER TABLE 兼容旧版本。
  - 加载代理时通过多轮 prepare 尝试不同列集，实现对历史版本的兼容。
  - 自动修复与重保存：当发现存储的代理清单与当前版本不匹配时，自动以新格式重写。
- 错误处理
  - 对于无效 UUID、损坏的 JSON/消息包进行跳过与告警。
  - 重复名称去重，保留首次出现的代理。
- 性能影响
  - 兼容路径与自动修复会带来额外的查询与写入操作，但仅在首次加载时发生。


### 组件二：会话存储（跨通道上下文与压缩）
- 设计要点
  - 使用消息序列化将会话持久化到 SQLite，并提供跨通道的统一上下文。
  - 支持会话列表、标签设置、删除与清理。
  - 提供"规范会话"（canonical session）用于跨渠道上下文聚合。
- 压缩与摘要
  - 当消息数量超过阈值时，生成摘要并截断旧消息，保持上下文窗口大小可控。
- 性能考量
  - 序列化/反序列化成本与磁盘 IO 成正比；建议合理设置阈值与窗口大小。

## 性能考量
- 初始化性能
  - 初始化为一次性操作，通常在启动时完成；建议在低负载时段执行。
  - 对大表执行索引创建时注意锁竞争，可在维护窗口进行。
- 存储与序列化
  - 会话与代理存储使用序列化，建议控制单条记录大小，避免过大对象。
- 查询优化
  - 知识图谱与会话查询应充分利用索引，避免全表扫描。
- 内存与 CPU
  - 内存整理周期性执行，建议调整衰减率与周期，平衡性能与效果。

## 故障排查指南
**更新** 迁移系统已移除，故障排查重点转向初始化问题。

- 初始化失败
  - 检查数据库连接权限与路径配置。
  - 查看具体表创建语句是否存在语法错误。
- 数据不一致
  - 校验结构化存储的兼容路径是否成功执行。
  - 检查代理清单是否被自动修复与重保存。
- 启动异常
  - 核对配置文件与环境变量，确认数据目录权限。
  - 使用健康检查命令诊断数据库连接与初始化状态。

## 结论
**更新** 迁移系统已完全移除，项目采用新的初始化策略。

OpenFang 现在采用"即时初始化"策略，通过直接创建所需表结构确保数据库在启动时即具备最新 Schema。结合内核启动集成与 CLI 引导，用户可以安全地完成系统初始化。由于迁移功能已移除，建议用户在升级前备份数据，并参考替代方案进行版本迁移。

## 附录

### 替代方案与最佳实践
**更新** 迁移系统已移除，提供替代方案：

- 替代方案
  - **手动备份与恢复**：升级前备份整个数据目录，升级后恢复数据。
  - **版本兼容性检查**：在升级前检查应用版本与数据库 Schema 的兼容性。
  - **渐进式部署**：先在测试环境验证，再在生产环境部署。
- 最佳实践
  - 在维护窗口执行系统升级，减少对业务的影响。
  - 对大表操作（如索引创建）分批进行，避免长事务锁。
  - 升级前后进行数据完整性校验（表存在性、关键字段、索引）。

### 数据完整性验证
- 验证清单
  - 核对关键表（agents、sessions、kv_store、memories、entities、relations、usage_events、canonical_sessions、paired_devices、audit_entries）是否存在。
  - 验证索引是否创建完成。
  - 对重要数据进行抽样查询与一致性检查。

### 初始化工具使用指南
- 命令入口
  - 使用 CLI 启动内核，内核自动执行初始化。
  - 首次运行时可通过交互引导选择"获取开始"，其中包含初始化选项。
- 诊断与健康检查
  - 使用健康检查命令查看初始化状态与数据库连接情况。
  - 如遇问题，查看日志并结合表结构定位失败位置。
